## 1. Verdict

**BLOCKED.** S3-01 must not open until the preconditions below are met.

The verdict deserves its precise reading. The Stage-3 design's *architecture* survived adversarial review: the command-catalog identities match Tier 0 exactly (nothing invented, nothing missing — independently derived before reading the spec's table), the authorization runtime order is source-backed against the real OutLabs package, the fence architecture is clean and conflict-free with the core `ClaimedJob` exclusion, the long-poll safety properties are correct, the runtime/embedded state machines are substantively sound, and the scope discipline is perfect (empty implementation diff across the whole range, live-reproduced test baseline). What blocks is a set of **documentation-completeness defects**: three BLOCKER findings where the frozen text requires the implementer to guess a security-relevant mechanism, credential path, or provisioning contract that fails against the real dependency; five HIGH findings of the same class; and two genuine Contract questions where adopted-base wire promises have no 0.1.2 backing (the exact class S3-CQ-01..03 existed to eliminate — two instances remained).

**No finding requires a SQL function, migration, grant, or architectural change.** Every remediation is a spec/ADR/authorization-doc amendment. Expected remediation size: one contract adjudication (both Contract questions fit a single additive protocol amendment on the ADR-015/016 pattern) plus one or two docs commits.

**Preconditions for opening S3-01:**

1. Adjudicate **R5-CQ-A** (list-jobs disposition) and **R5-CQ-B** (enqueue `created_at`) docs-first — recommended as one ADR (protocol document revision 1.0.4), both being manifest-wins normalizations of adopted-base promises, the same basis as ADR-015/016.

## 2. Findings

Severity-ordered. Each finding cites evidence, violated authority, a falsifiable construction, and the smallest contract-correct remediation plus test. File aliases: **SPEC** = `docs/Task Queue Stage 3 FastAPI and Authorization Specification.md`; **PROTO** = `docs/Task Queue Transport Protocol v1.md` (rev 1.0.3); **BASE** = `docs/design-review-2/03-protocol-draft.md` (adopted Tier-0 base); **MAN** = `docs/Task Queue 0.1 Function Manifest.md`; **AUTH** = `docs/Task Queue Authorization & Queue Permissions.md`; **OA** = the OutLabs authorization source checkout.

### BLOCKER

**R5-01 — Envelope/redaction ownership is unstated, and the stated integration's default failure path leaks the attempt fence.**
- Evidence: SPEC:298 (`create_router` returns an `APIRouter`), SPEC:303–305 ("FastAPI's default validation/error bodies never escape…"), SPEC:338–344 (the stable integration is only `compose_lifespans` + `app.include_router(router)`). Violated authority: BASE:50 (no raw validation bodies), BASE:137 (fence never in error detail), PROTO §2.3 (every response carries the envelope `request_id`), SPEC's own §5.1:139–142 (fence never in validation messages).
- A router cannot own `RequestValidationError` handling, unmatched-path 404, method-mismatch 405, or middleware-level 500 — those are application-level handlers the public integration never installs. Construction: `POST /taskq/v1/jobs/{id}/complete` with one missing/invalid sibling field → the framework's default 422 handler returns its native `{"detail": [...]}` body whose error entries carry the submitted `input` — **including the live attempt fence** — with no envelope and no `Taskq-Request-Id`. Every settlement body carries the fence, so every body-validation failure on a settle route is a fence-disclosure event under the default reading.
- Remediation: the spec must name the ownership mechanism — a custom route class that catches parsing/validation/handler/dependency exceptions per-route and renders envelopes with input echo suppressed, **plus** a mandatory prefix-level normalization step (mounted sub-application or documented required handler installation) for 404/405/500 — and state that mounting without it is unsupported. Test (S3-02): settle request with an invalid sibling field returns a `TQ422` envelope; assert the fence value appears nowhere in body, headers, or logs; repeat for unknown path, wrong method, and an authorizer that raises.

**R5-02 — Queue ensure is on the active always-on surface but executable only by the operator credential the ordinary pool must never hold.**
- Evidence: SPEC:110–111 (Queue bootstrap listed as its own active area, separate from Operator), SPEC:125–126 (only "Operator routes" are gated on the separate operator transport/pool), SPEC:287–289 (ordinary facade login "never operator"). Ground truth: MAN §5 (:397–405) and `src/taskq/sql/manifest.py:173` — `taskq.ensure_queue` is EXEC `taskq_operator`. Violated authority: ADR-011 §2 (facade login = producer+runner+observer+housekeeper, never operator); BASE:72 (action `admin`).
- Construction: the spec's advertised smallest deployment (one login, no operator pool) mounts the facade; `PUT /taskq/v1/queues/emails` is generated (not in the Operator area), executes on the ordinary pool, PostgreSQL denies EXECUTE → an opaque `TQ500` on a Protocol-active route, forever. The tempting fix — granting `ensure_queue` to the ordinary login — is exactly the privilege broadening the charter forbids. Independently discovered by two audit passes and source-verified.
- Remediation: reclassify queue ensure (and confirm `set_concurrency_limit`, already in the Operator area) into the operator-gated family: generated only when the operator transport is configured, executed via the observer+operator pool; and pin the authorizer rule for `admin`/`control` routes (operator routes require `operator_authorizer`, or explicitly use `authorizer` with a pool-only split — either is implementable, but it must be chosen docs-first). Test (S3-02): without operator config, `PUT /queues/{q}` is absent (surface reduction, not fallback); with it, `created|updated|unchanged` execute via the operator pool only; a pool-usage ledger proves the ordinary pool never ran an operator/admin SQL function.

**R5-03 — `provision_taskq_auth(roles="standard", mode="apply")` fails on its first run: `taskq-admin` grants a permission name the catalog never creates.**
- Evidence: AUTH §4.1 role table (:169) — `taskq-admin` grants **`taskq:*`**; SPEC §11.2:443–445 and AUTH §4.1:144 — the catalog is exactly "five global permissions plus five per distinct canonical queue" (no wildcard row). Ground truth (source-verified): OA `services/role.py` `_resolve_permissions_by_name` raises `InvalidInputError("One or more permissions do not exist")` for any missing name, reached from `create_role(permission_names=...)`; wildcard *matching* needs no row (`services/permission.py:559–590`), but role *assignment* resolves rows.
- Construction: fresh IAM database → seed 10 permission rows → `create_role("taskq-admin", permission_names=["taskq:*"], is_system_role=True)` → `InvalidInputError`, transaction rolled back. The Authorization doc's own §0 headline one-liner cannot succeed; SPEC S3-04's "role grants … are exact" is unfalsifiable as written.
- Remediation (choose one, docs-first): (a) when `roles="standard"`, seed one additional wildcard row `taskq:*` (legal under the real validator — a component may be exactly `*`) and amend the stated counts; or (b) define `taskq-admin`'s grant as the five explicit global names (authorization-equivalent under the any-of check; keeps "five global" honest). Test (S3-04): apply-mode first run on an empty IAM store creates all four standard roles.

### HIGH

**R5-04 — `reconcile=True` is unimplementable through the public role service for system roles.**
- AUTH §4.1:162 creates standard roles `is_system=True`; SPEC §11.2:452–459 promises reconcile convergence via "the public role service". Source-verified: every public mutation path hard-refuses system roles (OA `services/role.py` — `set_permissions_by_name`, add, remove, `update_role`, `delete_role` all raise "Cannot modify system role…"); only initial `create_role` attaches grants. A drifted or version-revised standard role can never converge without private-model writes (forbidden) or non-system roles (contradicts the doc). Remediation: create standard roles with `is_system_role=False` and amend AUTH §4.1, **or** spec `reconcile` as report-only-with-instructions for system roles. Fix the field name in the same pass — the real kwarg is `is_system_role`, not `is_system`. Test: S3-04 reconcile row exercises a drifted system role.

**R5-05 — "Personal keys never run workers" is not package-enforced in the deployed configuration.**
- AUTH §4.3/§4.4 and SPEC §11.1:438–439 present the personal-key prefix policy as the enforcement; source-verified: scope-policy validation returns before any check when the entity hierarchy is disabled (OA `services/api_key_policy.py:930` — `if not self.config.enable_entity_hierarchy: return`), i.e. enforcement exists only under EnterpriseRBAC — and AUTH §5:207 itself states "both hosts run SimpleRBAC". Construction: SimpleRBAC host, user holds a role granting `taskq:run`, mints a default personal key (empty scopes = no narrowing) → the personal key claims and settles jobs. S3-04's "personal-worker denial" vector passes only on an Enterprise fixture. Remediation: scope the claim to EnterpriseRBAC in AUTH §4.3/§4.4 (and the credentials sentence ADR-006 carries), state the SimpleRBAC residual honestly (denial holds via grant policy, not key policy), and pin the S3-04 vector's required configuration — one Enterprise-fixture denial vector plus one SimpleRBAC vector demonstrating the documented residual.

**R5-06 — The reserved queue-detail route must answer typed `TQ501`; the spec ships a native 405 by omission.**
- ADR-015 Decision 2: "A request receives the registered `TQ501` capability-inactive response"; PROTO §2.2 pins the negative capability vector. SPEC:118–120 says only "deferred out … no generated operation or client method" — with `PUT` mounted on the same path, the un-generated `GET` yields the framework's default **405** (native body, no envelope — compounding R5-01), violating ADR-015's explicit wire promise. The gated `GET /taskq/v1/workers` is handled correctly; the asymmetry is the defect. Remediation: distinguish "excluded from generation/OpenAPI/clients" from "unmounted" — the facade mounts a reserved-path responder returning the `TQ501` envelope, hidden from OpenAPI; add ADR-015's negative vector to S3-02.

**R5-07 — Remote-worker long polling composed with the unchanged Stage-2 claim loop breaks multi-queue latency and shutdown arithmetic.**
- SPEC:208–213 keeps Stage-2 supervisor semantics and sets `claim_wait_seconds` (default 25) on "the existing claim call". Ground truth (verified): `src/taskq/worker.py:439–496` round-robins queues with one blocking `transport.claim` per queue per sweep; `worker.py:384–390` `_stop` awaits the in-flight claim and never cancels it. Constructions: (a) three empty queues serialize up to 3×25 s per sweep — a job on queue B waits behind queue A's server-side wait, where Stage-2's listener gave sub-second wakeup; (b) SIGTERM during an in-flight 25 s claim blocks drain up to 25 s, and the spec's grace warning compares only `soft_stop_timeout`, so a "correctly configured" deployment hard-kills mid-settlement. Remediation: specify the composition — long poll only after a full empty sweep (or restrict `claim_wait_seconds > 0` to single-queue workers), and on stop either cancel the in-flight claim (accepting the already-specified unknown-outcome/lease-recovery path) or include residual `claim_wait_seconds` in the grace arithmetic and warning. Test: manual-clock scripted transport proving both the sweep-latency and stop-latency bounds.

**R5-08 — The facade-minted `timeout` claim outcome has no normalization target in the core model clients must return.**
- SPEC:183–184 (clients "return the same core typed domain values as SQL"), SPEC:316 (typed timeout). Verified: `src/taskq/protocol.py:196–201` `ClaimState` is `claimed|empty|paused|unknown_queue|unavailable` — no `timeout`; `timeout` is Tier-0 wire vocabulary (BASE:104) no SQL result produces. An unmapped state is an implementer guess consumed by `worker.py`'s claim loop. Remediation: one sentence — official clients normalize `timeout` to `ClaimState.EMPTY` (Tier 0 calls it "successful wait, no job"), elapsed/deadline data available on the HTTP result wrapper only; pin with a parity vector (SQL `empty` vs HTTP `timeout` normalize identically).

### MEDIUM

**R5-09 — Request-id invalid-case wire behavior is pinned only at Tier 3.** PROTO §2.3 defines valid and absent but for invalid says only "returns `TQ422`" — the envelope's `request_id` value for that very response, the header echo, and rejection-vs-authentication ordering exist only in SPEC §7.2 steps 1–3. ADR-016's own context says Tier-3 code cannot choose wire-contract decisions. Alternate protocol-conforming readings today: echo the invalid supplied bytes into the response header (header injection/log flooding), or reject pre-auth (an unauthenticated validation oracle). Remediation: one-sentence protocol errata under the adjudication ADR ("an invalid supplied correlation value is treated as absent for correlation selection — the response carries a minted UUID — and its TQ422 rejection is ordered after authentication"). Tests: oversize/non-ASCII header → TQ422 with minted UUID in envelope+header, supplied bytes nowhere in the response; same header with bad credentials → AUTH401 with minted id.

**R5-10 — Benchmark ID "B9" carries three conflicting identities.** PROTO §1 H-08 binds B9 to list-route keyset/EXPLAIN evidence; the harness doc §5 (:102) defines B9 = stats-under-depth (1M backlog) latency — and its open-constant table (:112) makes that identity load-bearing; SPEC §13 S3-AUDIT redefines B9 = generated client→ASGI→SQL path. The harness §5 heading also says "(B1–B11)" over a 13-row table. Failure: H-08's reactivation gate later "satisfied" by ASGI latency numbers instead of keyset EXPLAIN structure. Remediation: allocate B14 for client→ASGI→SQL in the harness, reference it from the spec, keep B9 = stats-under-depth, fix the heading.

**R5-11 — The Authorization doc contradicts the canonical wire, and T6 cites the stale matrix.** AUTH §2.1:52–54 lists enqueue queue-source "body (`queue` field)" and bulk "body — every distinct queue in the batch is checked"; §2.2:77 repeats it; §8 test 3 (:231) requires "Bulk enqueue spanning `emails`+`tools`". Canonical 0.1: bulk is path-addressed, one queue per call (BASE §3.2:94; SPEC §7.2:265 correctly says so). §2.1 also still lists the H-08-deferred `GET /jobs`, workflow/schedule rows, and a `tick` row against ADR-011.5. The harness T6 row imports "the queue-scoped matrix from the Authorization doc §8". Failure: a T6 vector that cannot be expressed against the canonical surface silently drops (coverage hole recorded green) or gets "fixed" by inventing a body-queue bulk route; a multi-queue legacy alias cannot honestly exist either (splitting a batch breaks one-atomic-batch). Remediation: amend AUTH §2.1/§2.2 to the 1.0.3 canonical shapes (body-queue rows marked legacy-alias-only), replace §8.3 with its canonical equivalent (batch on the `tools` path holding only `taskq_emails:enqueue` → 403 naming `tools`, nothing enqueued), align the `tick` row, re-point the harness citation.

**R5-12 — Checker-raised HTTP 429/503 have no protocol representation.** Source-verified: the real dependency surfaces 429 + `Retry-After` on API-key quota and 503 + `Retry-After` on auth-infrastructure loss (OA `dependencies/__init__.py:607–628`), both reachable through the adapter's checker call. The protocol pins only `AUTH401`/`AUTH403`; `TQ429` means queue backpressure and `TQ503` database/facade unavailability — mapping IAM throttling onto either lies about the cause; `TQ500` loses retryability. Remediation: pin the mapping in AUTH + the registry notes (recommended: auth-layer throttle → `TQ429` with `retryable=true`, honored `Retry-After`, `details.reason=auth_rate_limited`; IAM backing loss → `TQ503` with `details` naming the dependency class), with an S3-04 vector driving a scripted checker raising each.

**R5-13 — The auth session dependency's calling convention and lifecycle are unpinned.** The adapter sketch bare-awaits `self._session(request)`, but the real package's request-session providers are async generators, and the checker's signature accepts `session=` only when the auth object was constructed with its session integration (source-verified, OA `dependencies/__init__.py:1134–1165`, `core/auth.py:720–799`). The generic `QueueAuthorizer.authorize(request, action, queue)` carries no session, so the adapter must source and deterministically close one itself. Remediation: one paragraph in SPEC §11.1 — supported `session_dependency` shapes (awaitable, async generator, context manager), adapter-driven resolution and per-authorize close, and the requirement that the auth object expose the session parameter. Test: S3-04 adapter matrix runs against both an awaitable and an async-generator provider.

**R5-14 — Existence-hiding mode can leak by distinguishability.** SPEC §7.2:278–281 maps denial to 404 "without changing database behavior" but never requires the hidden 404 to be field-identical to the genuine `TQ001` missing-job envelope — a differing code, queue name, or `retryable` restores the existence oracle the option exists to remove. Remediation: one sentence — in hiding mode the denial response is field-identical to the TQ001 missing envelope modulo `request_id`; S3-02 asserts body equality of the two 404s.

**R5-15 — Catalog/provisioning input validation is looser than the queue grammar it serves.** The only pinned validation is "the real OutLabs validator", which lowercases silently, accepts hyphens, and has no length cap below the create-schema's 100 (source-verified, OA `utils/validation.py:187`, `schemas/permission.py:32–35`). `queues=["My-Queue"]` or `["Emails","emails"]` produce dead rows, silently normalized names, or wrong counts; "canonical queue" is undefined for catalog input. Runtime generation is safe (Tier-0/DB charset-pinned); only the provisioning input path is loose. Remediation: the catalog builder validates inputs against the taskq queue grammar (`[a-z0-9_]{1,57}`) and **rejects** — never normalizes; "distinct" = post-validation exact strings. Test: S3-04 catalog row rejecting a hyphened/uppercase queue.

**R5-16 — Client retry policy has no home in the single generated source.** SPEC §4.1's HTTP metadata list (method, path, action, queue source, state, adapters, status mapping) carries no retry/idempotency class, while §6 demands exact per-command retry behavior and PROTO bans hand-copied per-client tables. The core `ReplayRule` (FENCED vs STATE_DERIVED) cannot express "claim: never after unknown response" vs "enqueue: same non-empty key only" vs "non-keyed/mixed bulk: refuse". Remediation: add a per-command retry classification to the generated protocol table, generated into both clients, with a drift test against a Tier-0-derived §5-matrix oracle.

**R5-17 — "All H-09 bounds are checked before SQL" contradicts the ADR-012 diagnostic carve-out.** PROTO amendment 5: persisted job/attempt errors and cancel reasons are byte-truncated, never `TQ422`, "because diagnostic text must not prevent settlement". A naive facade rejecting a 3 KB failure diagnostic with `TQ422` blocks settlement over HTTP while direct SQL truncates and settles — a transport-parity divergence. Remediation: one carve-out sentence in SPEC §5.1 (diagnostic text passes through within the request ceiling; SQL truncation is authoritative) plus an S3-02 oversize-diagnostic vector proving truncate-and-settle parity on both transports.

**R5-18 — `ClaimWaitHub` channel lifecycle is unspecified and the Stage-2C notification source cannot express it.** Verified: `src/taskq/sql/notifications.py:41–67` fixes channels at `connect()` (re-connect refused while open; no add/remove API), but hub channels are per-queue and queues are dynamic. The spec neither states the subscription protocol (listen-on-first-waiter with UNLISTEN? listen-all-known with refresh on ensure?) nor acknowledges Stage 3 must extend the Stage-2C source — the charter's reuse-vs-second-machinery question left open. Remediation: specify channel management on the single listener connection, its reconnect re-LISTEN set, and the source API's dynamic channel operations; race test: waiter subscribes during listener reconnect, both orders.

**R5-19 — Client-disconnect observation is asserted but not mechanized.** SPEC:320–322 says disconnect cancels only the waiter, but a plain awaited endpoint is not reliably cancelled on disconnect unless the design observes the ASGI disconnect event. Without it, every dropped long poll's next deadline recheck executes an authoritative claim into a dead socket — converting a due job into a lease-length delay per drop. Remediation: require the wait loop to observe disconnect (concurrent await on the disconnect event / check before each re-claim) so no claim transaction begins after disconnect is observable; extend the S3-02 disconnect vectors with that assertion.

**R5-20 — R2-11 in an ASGI host has no named process-exit actor.** Stage-2's contract: a live sync handler is never released; `requires_process_exit` exists and the worker CLI is today's only actor (`cli.py` `os._exit(3)` path). SPEC §9.3/§10 preserve the requirement but never designate who, in a web process, monitors the flag, flips readiness, and terminates. Without an actor: shutdown hangs forever or a healthy-looking web process hosts an abandoned side-effecting thread. Remediation: specify — the runtime monitors `requires_process_exit`, marks failed/unready, stops intake, and terminates the process as the CLI does (documented and acknowledged alongside process multiplication). Test: the Stage-2 abandoned-sync fixture inside a live ASGI lifespan.

**R5-21 — The catalog oracle's independence is contradicted by the spec's own sentence.** SPEC §4.1: "That one source generates or drives: … a catalog test independently derived from the Tier-0 Protocol and Function Manifest." If the one source generates it, it is not independent — an implementer may legally generate the "oracle" from `taskq.protocol` and neuter the drift gate. Remediation: reword — the catalog test is hand-derived from PROTO/MAN, never imports generated metadata for expected values, and the deterministic generation artifact is compared against it. This is also what makes the human-written HTTP metadata acceptable rather than a second human-owned protocol.

**R5-22 — Per-queue stats for an unknown or not-yet-snapshotted queue has no defined outcome.** BASE:173 defines only `ok 200`; `get_queue_stats` returns the empty set for an absent key and the snapshot lags queue creation by up to a tick. `TQ001` vs empty-with-`as_of` is an implementer guess with observable consequences (a queue created seconds ago 404s). Remediation: the spec picks one (recommended: document snapshot-lag empty semantics — empty result with `as_of`, `TQ001` reserved for the queue absent from configuration) and pins it in the catalog oracle plus a vector.

## 3. Contract questions

Cited without code workarounds, per charter. Both are adopted-base promises with no 0.1.2 backing — the S3-CQ-02/03 class — and both fit one additive ADR + protocol document revision 1.0.4.

**R5-CQ-A — General job list has no adjudicated 0.1 disposition; the Tier-0 texts cannot all be satisfied.**
Irreconcilable passages: PROTO §2 amendment 3 ("the base's §3.6 list-jobs row is downgraded to 'operator-minimal, pre-H-08' for 0.1" — implying an active reduced form) vs PROTO §3 (:136) ("H-08 … explicitly deferred **with capability gates**" — implying a reserved `TQ501` path) vs MAN:3 + §7 (:513) ("`list_jobs` beyond the operator-minimal form" is absent — implying an operator-minimal form exists) vs migration reality (`src/taskq/sql/migrations/0001_initial.sql:87`: "**list_jobs: absent**"; no list function or view of any form exists in the 0.1.2 catalog, and T2 asserts catalog-manifest identity). PROTO §2.2 — created by revision 1.0.2 as the visible deferral registry — omits the route entirely. The Stage-3 spec resolves it correctly in substance ("General job list is H-08-deferred"; S3-01: "no generated method") — but a Tier-3 resolution of Tier-0 ambiguity is the exact pattern S3-CQ-01..03 existed to eliminate, and an implementer reading amendment 3 + MAN §7 in good faith builds an unbacked "operator-minimal" route whose only reachable backings are privilege broadening or wire invention.
Recommended adjudication: under ADR-016's reusable rule this command is **undesigned** (H-08 holes: cursor tuple, projection, indexes) → **deferred out**. Add `GET /taskq/v1/jobs` to the §2.2 deferred table with the H-08 reactivation gate and a pinned 0.1 disposition; correct MAN §7's wording to "no `list_jobs` exists in 0.1"; strike or footnote amendment 3's "operator-minimal" phrase. One negative catalog vector pins the reserved path, mirroring ADR-015's.

**R5-CQ-B — The enqueue response's promised `created_at` is unproducible from contract 0.1.2.**
Irreconcilable passages: BASE §3.2 (:84) — enqueue `created` response data is "`job_id`, queue, job_type, **created_at**" — vs MAN/`src/taskq/sql/manifest.py:172` — `taskq.enqueue(...)` returns `TABLE(job_id uuid, created boolean)`, nothing more (verified; the core `EnqueueResult` accordingly has no `created_at`, and its `queue`/`job_type`/`idempotency_key`/`scheduled_at` are command-echo, not row truth — on `existed` they report the *requested* values, not the existing row's). No honest facade source exists: a follow-up `get_job` read would mix observer capability into a producer route and double the round trip; a facade-minted timestamp would be wire invention; a contract change is out of Stage-3 scope by the spec's own boundary.
Recommended adjudication: manifest-wins amendment (the ADR-016 queue-ensure-version basis) removing `created_at` from the enqueue response rows; clients needing it read `GET /taskq/v1/jobs/{id}`. The same amendment should require generated wire models to distinguish row-truth fields from request-echo fields so `existed` responses cannot present requested values as stored state, and the catalog oracle must assert response-field sets per command so this class is caught mechanically (this also disposes of the wire-side half of finding R5-08's neighbor class).

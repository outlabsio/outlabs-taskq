# Gap Analysis: diverse-data-api vs qdarteAPI Postgres Task Queues

> **Provenance:** Produced 2026-07-06 by a cross-repo audit of the Postgres task queues in `diverse-data-api` (`scrape_jobs` queue domain) and `qdarteAPI` (`qdarte_ops.worker_jobs` platform). The input reports cited by code (DC, DCP, QC, QO, PA, gap_analysis) were working artifacts of that analysis session; all `file:line` citations refer to the two repos as of this date. Identical copies of this document live in both repos' `docs/` folders — treat them as one spec. Vault index note: `lifeOS/Projects/Business/QDarte/Postgres Task Queue Cross-Repo Audit & Unified Design.md`.

Date: 2026-07-06. Inputs: `diverse_core.md`, `diverse_control.md`, `qdarte_core.md`, `qdarte_orchestration.md`, `prior_art.md` (same scratchpad directory). All file:line citations below are quoted from those reports; "DC" = diverse_core, "DCP" = diverse_control, "QC" = qdarte_core, "QO" = qdarte_orchestration, "PA" = prior_art.

Both systems are descendants of the same design (the Diverse queue explicitly adapts "patterns from the Qdarte Unified Postgres Task Platform", DC §0), so this is a comparison of two forks of one idea: Postgres-as-queue, `FOR UPDATE SKIP LOCKED` claim, lease + heartbeat + fencing token, attempt ledger, event audit trail, HTTP worker transport, at-least-once delivery. The differences are in schema rigor, retry sophistication, idempotency enforcement, orchestration, cleanup discipline, and control-plane maturity.

## 1. Capability matrix

Legend: **strong** / **adequate** / **weak** / **missing**. Industry column = the consensus of the message-queue extension, the Go/Postgres job queue, the Node/Postgres SQL-first worker, the Node/Postgres queue library, the Rails-native Postgres queue, the Elixir/Postgres job framework, the mature Python/Postgres task library, a lean Redis/asyncio job library per prior_art.md.

| Dimension | diverse-data-api | qdarteAPI | Industry best practice |
|---|---|---|---|
| **Schema design** | **Adequate** — real Postgres enums for statuses, FK constraints, purpose-built partial claim/lease/idempotency indexes (DC §1.1); but naive `TIMESTAMP WITHOUT TIME ZONE` everywhere (DC model_base.py:19-21), a no-op "safety net" index (DC §11.1), and missing hot-path indexes (DC §11.8). | **Weak** — zero FK constraints, zero CHECKs, statuses are free strings (QC §1, F17); redeeming features: timezone-aware timestamps, UUIDv7 time-ordered PKs, per-table autovacuum tuning (QC §1.1); claimable partial index exists only in the migration, not the model (QC §1.1). | **Strong** — partial index exactly matching the claim predicate, fillfactor 80-90 for HOT updates, per-table autovacuum overrides, DB-enforced invariants (PA §2.1, §4.1, §4.10). |
| **Claim / delivery semantics** | **Strong** — three-tier SKIP LOCKED claim (jobId → affinity → platform FIFO) backed by partial indexes; fencing via `current_attempt_id`; at-least-once, honestly documented (DC §2.3, §3). | **Strong** — SKIP LOCKED claim + `_lock_running_job_attempt` double row-lock + `_compare_and_set_running_job` fenced UPDATE, the strongest fencing of the two (QC §2.2-2.4); stress harness: 0 duplicate claims at 80 jobs/s (QC §3). | **Strong** — same primitive everywhere; ideal form is one atomic `UPDATE ... RETURNING` over a SKIP LOCKED subselect (the message-queue extension/a lean Redis/asyncio job library shape, PA §4.1). |

| **Retries / backoff** | **Adequate** — retry-as-same-row via `scheduled` status; but fixed 5m/30m/2h ladder, no exponential, no jitter (DC §4); `retry_after_hint` parser silently rejects the documented int/ISO formats (DC §11.7); `max_retries` actually means max attempts and releases burn the budget (DC §11.6). | **Strong** — per-job-type registry policies (max_attempts, fixed/exponential, cap), hint parser accepts seconds/ISO/duration, worker-supplied `retryable` mined from results (QC §4); gaps: no jitter, lease-expiry path bypasses the whole machinery (F6), unregistered job type in retry path → KeyError 500 (F7). | **Strong** — exponential + jitter (the Elixir/Postgres job framework: exp + 15s pad + jitter), high default budgets (20-25), snooze-without-consuming-attempt (the Elixir/Postgres job framework `{:snooze, s}`), backoff as data on `scheduled_at` (PA §4.4). |
| **Dead-lettering** | **Weak** — terminal `failed` rows are the DLQ; no redrive endpoint; operator cancels are conflated into `failed`, polluting failure stats (DC §2.12, §11.11; DCP 7.8). | **Weak** — same terminal-rows-as-DLQ; dependents auto-canceled; a recovery lane exists but defaults off (`exhausted_recovery_limit: 0`, QC §4); no manual retry endpoint (QO §5.1). | **Adequate is enough** — a dead-letter *state* kept for inspection beats a DLQ *table* unless redrive tooling exists (the Node/Postgres queue library has real DLQs + redrive; the Go/Postgres job queue `discarded`; PA §4.4). |
| **Priority / ordering** | **Adequate** — int priority (lower wins) + created_at FIFO, index-backed; affinity_key and job_kind routing on top (DC §5); job_kind not in the claim index (DC §11.8). | **Adequate** — same ordering with named constants (URGENT=0…BACKGROUND=500); but priority semantics are inconsistent across call sites — at least one handoff dep rioritizes while reading like a boost (QO 7.8); claimable index shape doesn't serve the sort (F10); no id tie-breaker. | **Strong** — numeric priority + covering index that serves the exact claim sort (the Rails-native Postgres queue); deterministic tie-breaker on id (PA §1.5, the Go/Postgres job queue). |

| **Dedup / idempotency** | **Strong** — partial UNIQUE index on `idempotency_key WHERE status NOT IN (terminal)` + bulk `ON CONFLICT DO NOTHING` (DC §1.1, §2.2); blemishes: single-enqueue path is SELECT-then-INSERT and 500s on the race (DC §11.4), CSV-import dispatch enqueues carry no keys (DCP 7.7). | **Weak** — `ix_worker_jobs_idempotency_key` is **not unique**; check-then-insert everywhere; concurrent enqueues duplicate jobs and side effects (F1, QO 7.2); JSONB-path entity dedup has no supporting index (QC §3). | **Strong** — index-enforced beats query-enforced, always: the Go/Postgres job queue args-hash partial unique index, the mature Python/Postgres task library `queueing_lock`; insert with `ON CONFLICT` and report whether insertion happened (PA §2.6, §4.5). |
| **Batching** | **Adequate** — bulk enqueue ≤1000/call with conflict handling (DC §2.2); batch unit is inside the job (counties/requests arrays); claim is strictly LIMIT 1 per HTTP call (DC §5). | **Weak** — no bulk enqueue; claim LIMIT 1; worker "burst mode" (immediate re-claim after success) is the only compensation (QC §5); hardening item P2-3 (batch claim) skipped. | **Adequate/strong** — batch claim (`LIMIT N`) reduces hot-row contention; the Go/Postgres job queue uses `COPY FROM` for bulk insert (PA §2.2, §1.2). |
| **Per-type / per-key concurrency control** | **Missing** at claim time — caps live only in fleet size (`worker_pool_targets` desired/soft/hard), "a platform's cap is only enforced by how many workers you run" (DCP §3.9, 7.8). | **Adequate** — `worker_concurrency_pools.max_in_flight` enforced in-DB at claim with `FOR UPDATE` on the pool row + derived count (no drift) (QO §2.2); but cooperative-only (non-pool workers bypass it), pool≠job-type binding, and a lock convoy at scale (QO 7.4, F8, F9). | **Strong** — a lean Redis/asyncio job library `group_key NOT IN (active)` anti-join (cheapest per-key mutex); the Rails-native Postgres queue semaphore rows with block/discard; per-queue worker concurrency (PA §1.5, §1.8, §4.8). |
| **Rate limiting / backpressure** | **Missing** — `retry_after_hint` is the only backpressure input; no admission control (DC §5); audience-sync reads run job-counts for its own backpressure (DCP §3.9). | **Weak/adequate** — LLM-lane suppression is a real DB-backed admission filter per provider lane (QO §2.5); nothing generic; no global pause. | **Adequate** — mostly a paid feature (the Elixir/Postgres job framework Pro); practical baseline is bounded batch/concurrency + admission filters (PA §2.7). |

Having one *coherent* dependency layer already exceeds OSS baseline; having three incoherent ones does not. |

## 2. Strengths worth keeping from each system

### From diverse-data-api

1. **DB-clock everywhere.** Every mutating method reads `select(func.now())` and normalizes it; leases never use the app clock (DC service.py:158-163). This is exactly the hardening item qdarte skipped (QC F4). Non-negotiable for the unified design.
2. **Index-enforced idempotency + race-safe bulk enqueue.** Partial unique index over non-terminal statuses, and `INSERT ... ON CONFLICT DO NOTHING` with the matching `index_where` predicate, `RETURNING` to report exactly what landed (DC §1.1, service.py:611-654). The correct pattern per PA §2.6 — qdarte has nothing like it.
3. **Scheduler windows.** `UNIQUE(platform, window_key)` where the key embeds the deterministic fire instant, claim-via-`ON CONFLICT ... RETURNING`, stale-claim takeover, settle with ownership check (DCP §2.1). This is the leaderless-cron pattern PA §4.9 recommends (the mature Python/Postgres task library), independently arrived at. Keep the table; replace the weak client-side due-detection.
4. **Self-healing claim.** Reaper + scheduled-promotion run inside every claim transaction (DC service.py:668-669) — no dependency on a separate liveness process for the core recovery path (though it must move off the hot path at scale; see gaps).
5. **Fail-with-partial-progress semantics.** `fail_job` persists partial results, rewrites `counties` to the remainder so retries only redo what's left, and a taxonomy of attempt outcomes (`retry_exhausted`, `nothing_remaining`, `canceled_after_expiry`...) (DC §2.8). Rich, honest failure modeling.
6. **Client-side settle idempotency.** The worker client retries complete/fail on 5xx and resolves a post-retry 409 by re-fetching the job and accepting expected terminal states (DCP §3.5, queue.py:135-151) — the only mitigation either system has for the non-idempotent-complete problem.
7. **Run/dispatch control plane.** Durable `queue_runs` with their own lease + cursor + heartbeat (a second-level claim loop), scope-key dedup for delivery runs, cancel that cascades `cancel_requested_at` to all child jobs, worker pool targets with generation-fenced reconciliation, drain-as-sugar (DCP §1). The most mature ops layer of the two.
8. **Environment safety interlocks.** Workers refuse a production API without `ALLOW_PRODUCTION_API=true` and refuse env mismatches (DCP §3.1). Cheap, effective, worth standardizing.
9. **Hard-won defensive details:** event-message truncation ("An overflow here used to 500 the whole fail call, leaving the job leased and the queue wedged", DC service.py:225-230); NOTIFY emitted in-transaction so rollbacks never produce phantom wakeups; heartbeat response doubling as the cancel channel.

### From qdarteAPI

1. **The CAS fencing discipline.** `_lock_running_job_attempt` (lock attempt AND job, verify triplet) + `_compare_and_set_running_job` (`UPDATE ... WHERE status='running' AND current_attempt_id=:a`, rowcount must be 1) on every terminal/requeue path (QC §2.3-2.4). Strictly stronger than Diverse's lock-then-check; it protects even against direct-SQL writers. Seed the unified design with this.
2. **Registry-driven per-type execution policies.** `JobExecutionPolicy` with priority, lease, and a real `JobRetryPolicy` (fixed/exponential, base, cap) resolved per job type at enqueue and retry time (QC §4). Diverse's one global 5m/30m/2h ladder should die in favor of this.
3. **Retry-hint parsing done right.** Seconds, ISO-8601, and duration grammar all accepted (QC jobs.py:5234-5264) — the exact contract Diverse promised and failed to implement (DC §11.7).
4. **Waiting/queued split + dependency edges + workflow runs.** A `waiting` status distinct from `queued`, `worker_job_dependencies` with satisfied_at, promote-on-unlock, cascade-cancel of dependents, workflow status rollup (QC §2.9; QO §3.1). The right *primitives* for chaining, even though production usage fragmented.
5. **Concurrency pools.** DB-enforced `max_in_flight` with a derived (drift-free) in-flight count, serialized on the pool row (QO §2.2). The mechanism ("six LLM workers, one in-flight LM Studio call") is exactly what per-resource caps need; fix the cooperative-only binding, keep the design.
6. **The janitor + retention discipline.** Tiered event pruning (verbose event types die at 7d), terminal-job deletion with full cascade, artifact TTL table so bulky output never lives on the job row, workflow cleanup, and the documented backup-before-cleanup ordering (QC §9). Plus per-table autovacuum overrides in a migration (QC §1.1) — the only bloat engineering in either system.
7. **Hybrid LISTEN/NOTIFY done to spec.** NOTIFY in-transaction (commit-gated), one channel, listeners keep a fallback poll, degrade to polling on connection failure (QC §6) — matches PA §2.4/§4.7 consensus exactly.
8. **UUIDv7 keys** — time-ordered, append-mostly PK b-tree (QC §1), better than random UUIDs for a high-churn table.
9. **Advisory-lock + state-row coordination for the platform tick** (`pg_try_advisory_xact_lock` + `platform_control_state` min-interval, QC §6.2) — a clean leaderless dedup for opportunistic maintenance, and the CLI escape hatch (`qdarte-platform-control queue-tick|janitor|daily-ops`) for real cron.
10. **The hardening-audit practice itself.** A written audit (worker-queue-hardening-plan.md) whose P0 items are all demonstrably implemented and whose skipped items map one-to-one onto today's residual bugs (QC §11). Process worth keeping regardless of code.

## 3. Shortcomings & gaps, ranked by severity

### Severity 1 — active correctness bugs (lost work, duplicate jobs, wedged systems)

1. **[qdarte] Idempotency is not enforced by the database.** Non-unique index (`models.py:48`, migration 0012:81-86) + SELECT-then-INSERT (`jobs.py:2747-2763`, 3197-3202). Concurrent enqueues — which the architecture *guarantees* via overlapping orchestration layers (QO 7.1) — create duplicate jobs and duplicate external side effects (LLM spend, publishes). QC F1, QO 7.2. Industry verdict: "Index-enforced beats query-enforced: no TOCTOU race" (PA §2.6).
2. **[qdarte] Lease-expiry requeue bypasses backoff and the attempt budget.** `_reclaim_expired_running_attempts` CAS-requeues straight to `queued` with no `_schedule_job_retry`, no backoff, no `max_attempts` check (`jobs.py:4999-5024`); a job that always dies by lease expiry loops **forever, immediately, with a NOTIFY stampede each time** (QC F6). The inverse asymmetry also holds: lease expiries count toward `attempt_count`, so 5 expiries + 1 transient failure = permanent `failed`.
3. **[diverse] Maintenance will wedge on FK violations — and nothing runs it anyway.** `run_maintenance` deletes `scrape_jobs` after clearing only 3 child tables, but ≥7 other columns FK into `scrape_jobs.id` with no ON DELETE (`prospects.last_job_id` is stamped on every auto-chain candidate — chain_enqueue.py:241, migration a1b2c3d4e5f6:69; render_jobs.*, letter_batches, worker_presence...). First 30-day-old referenced job → FK violation → whole-transaction rollback → oldest-first selection retries the same poisoned batch forever (DC §11.2). Compounding: no cron/task/worker invokes maintenance at all (DC §11.3, DCP 7.4) — unbounded table growth is the steady state.
4. **[diverse] One failed heartbeat permanently kills lease extension → guaranteed lost work.** `HeartbeatLoop._run` returns on first exception (worker.py:697-700); a transient API blip mid-batch → lease expires → reaper reschedules → original worker finishes and its complete 409s against a `scheduled` job, which the 409-recovery does not accept → **the finished batch's results are discarded and the whole job re-runs** (DCP 7.2).
5. **[qdarte] Orphan-detection can yank live jobs.** `ps`-based process discovery cannot see host launchd workers from inside the API container (the documented production layout); any running attempt invisible to `ps` with heartbeat older than 30s (`ORPHANED_RUNNING_GRACE_SECONDS`, runtime_control.py:39) is force-reclaimed while the worker is mid-LLM-call → duplicate external side effects (QO 7.3). State stays consistent (CAS), money and side effects do not.
6. **[diverse] Mid-transaction `session.rollback()` in the auto-chain discards caller work.** `chain_enqueue.py:313-315, 289-295` roll back the *shared ingest transaction* on duplicate-key flush, then continue as if nothing happened — uncommitted ingest writes silently vanish (DC §11.5). Needs SAVEPOINT (`begin_nested()`).
7. **[qdarte] GET endpoints commit writes mid-transaction.** `list_jobs`/`get_job_detail` run the reaper and `session.commit()` if anything reclaimed (jobs.py:4820-4822, 4853-4855); `get_job_detail` is called at the tail of claim/complete/fail/release — a hidden partial-commit that releases row locks before the enclosing service method finishes, and makes monitoring dashboards de-facto reapers with untraceable actor identity (QC F2, F3).
8. **[both] Worker keeps executing after losing the lease; complete is not idempotent.** Diverse: DCP 7.2; qdarte: heartbeat failure only logs (loop.py:140-141), nothing aborts the handler (QC F5). Both are at-least-once by design, but neither closes the loop by (a) aborting on heartbeat 409 or (b) making complete idempotent for network retry (DC §3: a lost complete response is indistinguishable from theft).

### Severity 2 — correctness-adjacent design flaws

9. **[diverse] `max_retries` is really "max attempts" and releases/drains burn it.** `attempt_count` increments at claim; release does not exempt it; N compose-driven drains + 1 real failure terminal-fails a healthy job as `retry_exhausted` (DC §11.6). qdarte has the identical flaw for releases (QC §2.7) plus the dead `skip_own_releases` flag, so a handler-less worker can burn a job's entire budget in claim/release spin (QC F12).
10. **[diverse] Single-enqueue idempotency race returns 500** instead of the existing job — the bulk path handles it, the single path was never retrofitted (DC §11.4, service.py:490-524).
11. **[qdarte] Unregistered job type in retry path → unhandled KeyError → job stuck running until lease expiry** (F7, jobs.py:5279; the codebase demonstrably retires job types).
12. **[diverse] Rendering reset bypasses the state machine** — force-writes `status=failed` without finishing the attempt or checking live ownership, leaving dangling `running` attempt rows and using app clock (DC §11.10).
13. **[qdarte] Janitor deletion of dependency edges can un-gate dependents:** deleting a 30-day-old terminal job converts a forever-blocked `waiting` dependent into "run now" (F15). Sibling risk: >50 permanently-waiting rows starve the promotion window (F14).
14. **[diverse] Dispatch-lease takeover crashes the losing dispatcher and duplicates CSV jobs** (no idempotency keys on CSV-import enqueues; 409-in-except-handler kills the process) (DCP 7.7).
15. **[qdarte] publish_scope executes business side effects inside the completion transaction** — the API is a hidden executor; a slow publish holds job/attempt row locks and couples ledger latency to content work; failure amplifies into full job re-execution (QO 7.9, QC F13).
16. **[diverse] `uq_scrape_jobs_running` is a no-op** (`UNIQUE(id) WHERE status='running'`, id is the PK) sold as a "database-level safety net" in code comments and the design doc (DC §11.1). The real guard is on the attempts table; the misleading one costs writes and trust.
17. **[both] Coarse trust model.** One shared `job:write` credential for the entire fleet + admin UI; fencing trusts client-supplied `worker_id`/`attempt_id`, and attempt ids are readable via GET (DCP 7.11). qdarte additionally accepts client-supplied timestamps for heartbeat bookkeeping (QC F11).

### Severity 3 — scaling and bloat risks

18. **[diverse] Zero bloat engineering.** No autovacuum tuning (promised in the design doc, never shipped — DC §0), no fillfactor, no scheduled cleanup, `job_results.job_id` unindexed (seq-scan on the fastest-growing child table), no `completed_at` index for the stats windows or terminal cleanup, no partial index for scheduled promotion (DC §11.8). Prior art calls hot-table churn the #1 killer (PA §2.1); the design doc itself agrees and then nothing was done.
19. **[both] Housekeeping on the claim hot path.** Every claim — including every idle poll of the whole fleet — first scans/locks up to 100 (diverse) / 50 (qdarte) expired + scheduled jobs across all lanes, with per-waiting-job dependency counts on qdarte (DCP 7.5; QC F19). N idle workers × poll rate = continuous cross-lane write amplification; PA's answer is a small dedicated tick (the Go/Postgres job queue scheduler every 5s) or claim-predicate-as-sweeper (the message-queue extension §4.2).
20. **[qdarte] NOTIFY stampede + pool convoy.** Every NOTIFY wakes every listener regardless of job type (loop.py sets the event unconditionally, F19); all claims in a concurrency pool serialize on one `FOR UPDATE` row held across the entire claim transaction (F9).
21. **[diverse] Queue on pg_duckdb with `duckdb.execution=on` globally** — OLTP queue on an analytics-extension substrate; qualifying SELECTs may route through the DuckDB executor; nothing pins it off for queue sessions (DC §11.12). Also 4 uvicorn workers × (8+4) pool conns vs max_connections=120 shared with everything (DC §6).
22. **[diverse] Hot-path write amplification in the control plane:** `_ensure_worker_control_defaults` re-SELECTs full seed sets on every presence heartbeat (DCP 7.15); `_platform_stats` runs ~9 queries × ~15 platforms per dashboard refresh (DC §11.13); mid-job presence heartbeats null out pool identity making `onlineCount` flap and skipping scale-down checks (DCP 7.3).
23. **[qdarte] `complete_job` mega-transaction** — publish + proxy health + artifacts + pipeline transitions + follow-up enqueues + dependency unlock + workflow refresh in one transaction (F13); plus a redundant post-CAS double-write of the same row (F18) on the hottest write path the autovacuum tuning is trying to protect.
24. **[qdarte] O(missed-intervals) recurring catch-up in Python inside the tick transaction** (F16).

### Severity 4 — operational gaps

25. **[both] No pause/resume/drain of intake.** Diverse: no per-platform claim pause, no operator retry/requeue/purge (DCP 7.8). qdarte: no global pause; incident response means touching specs + pools + lane suppression separately (QO §5.4).
26. **[both] No metrics/alerting.** Diverse has stats endpoints but no exporter/alerts (Grafana is a plan doc, DCP §6.4); qdarte has Prometheus wired but exports zero queue metrics — "alerting on 'queue not draining' is currently impossible without SQL access" (QC F21).
27. **[diverse] Scheduler loses windows across midnight, single timezone, no cron grammar, single replica** (DCP 7.9); stale-claim takeover at 900s can race slow enqueues (DCP 7.10).
28. **[diverse] NOTIFY without listeners for two platforms; chain doc claims "no polling" while workers poll at 30s** (DC §11.9). **[qdarte] Two worker frameworks + dead contract fields** (QO 7.10, F12). **[diverse] Contracts hand-mirrored across repos, mixed camelCase/snake_case** (DCP 7.14).
29. **[diverse] Cancel conflates with failure** — no `cancelled` job status; operator cancels pollute failure rates (DC §11.11). qdarte gets this right with a first-class `canceled` status.
30. **[diverse] Doc drift:** models.py points to a design doc that doesn't exist in the repo (DC §0); **[qdarte]** claimable index exists in migration but not model metadata — test-built schemas silently differ (QC §1.1).

## 4. Divergences a unified standard must reconcile

1. **Status vocabulary and state machines.**
   - Diverse jobs: `pending / scheduled / running / completed / failed / partial` — no cancelled status, `scheduled` doubles as both delay and retry state, `partial` is a distinct terminal (DC §2).
   - qdarte jobs: `waiting / queued / running / completed / failed / canceled` — first-class `canceled`, `waiting` covers deps + delay + recurring templates + retry backoff (QC §2).
   - Industry: the Go/Postgres job queue `available/scheduled/pending/running/retryable/completed/cancelled/discarded` — separate `retryable` and `discarded` states (PA §1.2).
   - **Reconcile as:** qdarte's set + a distinct terminal for exhausted-retries (`discarded`-style), and decide whether Diverse's `partial` survives as a status or becomes result metadata (recommendation: metadata + outcome, not a status — it complicates every terminal check today, e.g. the idempotency-predicate bug DC §1.1).
2. **Attempt statuses differ:** Diverse `running/completed/failed/released/reclaimed` vs qdarte `running/completed/failed/abandoned` + outcome strings. Unify on one enum + one outcome vocabulary.
3. **Retry semantics.** Diverse: retry = flip to `scheduled` with fixed ladder, `max_retries` misnamed. qdarte: registry policy + `waiting` + schedule row + CAS. Must pick one mechanism (recommendation: qdarte's policy engine + Diverse's DB-clock + explicit "does this attempt consume budget?" flags for release/lease-expiry/snooze — neither system has that today, both need it; see gaps 2 and 9).
4. **Naming.** `scrape_jobs/platform/job_kind/affinity_key/max_retries/scheduled_for/error` vs `worker_jobs/job_type/scope_kind/scope_key/max_attempts/scheduled_at/error_summary`; `queue_runs` vs `workflow_runs`; `worker_pools`(machines) vs `worker_concurrency_pools`(admission caps) — same word, different concepts; priority default 2 (0-10) vs 100 (0-1000). Every one of these needs a single canonical term; "pool" is the most dangerous collision.
5. **Clock authority.** Diverse: DB `now()` for all lease math. qdarte: `datetime.now(UTC)` app clock (F4). Unified: DB clock, full stop.
6. **Schema philosophy.** Diverse: enums + FKs + DB constraints (but naive timestamps). qdarte: no FKs, string statuses, app-enforced integrity (but timestamptz + UUIDv7 + autovacuum). Unified: take both halves — FKs *with explicit ON DELETE rules* (Diverse's missing-ON-DELETE is what wedges its maintenance), CHECK/enum statuses, timestamptz, UUIDv7, autovacuum + fillfactor, and every index declared in model metadata AND migration.
7. **API-vs-SQL access.** Both are "HTTP-only, except...". Diverse's exceptions: in-API list-delivery worker + chain-enqueue direct writes. qdarte's: direct LISTEN connection + the API itself executing publish work. Unified standard must pick a rule: HTTP-exclusive for external workers, direct service-layer access allowed only in-process through the same service class (never raw SQL), one shared contracts package (kill the hand-mirrored Pydantic models, DCP 7.14).
8. **Execution topology.** Diverse: platform lanes (one queue table, `platform` column), pull workers per platform, dispatcher-expanded runs, compose+reconciler fleet, scheduler process. qdarte: job-type lanes, `supported_job_types` claim filter, fleet specs + launchd/docker backends, no scheduler daemon (self-tick + CLI cron). Reconcile: lane key (`platform` vs `job_type` — recommendation: one `queue`/`lane` column + `job_type`, per the Node/Postgres queue library/the Rails-native Postgres queue), and one fleet-control story (Diverse's generation-fenced pool targets are cleaner than qdarte's four coexisting runtime-control backends).
9. **Orchestration model.** Diverse `queue_runs` (batch grouping + dispatch lease) vs qdarte `workflow_runs` + dependencies + handoffs + launch pipeline. These solve different problems (batch-of-N vs DAG) and a unified design likely needs both, but as *one* layer each — not qdarte's three chaining idioms (QO 7.1) or Diverse's split run-settlement paths (DCP 7.6).
10. **Cron.** Diverse: server window table + client YAML schedules, no catch-up. qdarte: recurring template rows, no cron grammar, O(n) catch-up. Unified: cron grammar + Diverse's window-claim table (its `(platform, window_key)` UNIQUE is the leaderless-dedup pattern PA §4.9 endorses) + explicit bounded backfill policy.
11. **Cleanup.** Diverse: unscheduled + FK-broken. qdarte: scheduled, tiered, backed up. Unified: qdarte's janitor design, made cascade-safe by FKs with ON DELETE rules instead of hand-maintained deletion order (QC F17).

## 5. Verdict

**qdarteAPI is closer to the right foundation for the core engine; diverse-data-api is closer to the right foundation for everything around the engine.** A unified design should be qdarte's kernel operated Diverse's way.

Why qdarte's kernel wins:

- Its ownership model (double row-lock + CAS-fenced UPDATE, QC §2.3-2.4) is the strongest correctness core in either system and is the direct product of a written hardening audit the code demonstrably implements — validated by a stress harness (0 duplicate claims, QC §3, §11). Diverse's equivalent is good but weaker (lock-then-check, no CAS on the UPDATE), and its advertised job-level DB backstop is a no-op index (DC §11.1).
- Its lifecycle vocabulary is more complete (first-class `canceled`, `waiting` vs `queued`) — Diverse must bolt cancellation onto `failed` and overloads `scheduled` (DC §11.11).
- Its policy layer (per-type registry: priority/lease/retry with exponential backoff and working hint parsing) and its operational hygiene (janitor, autovacuum, artifact TTL, backup-before-cleanup) are exactly what prior art prescribes (PA §4.3-4.4, §4.10) and what Diverse promised in its design doc but never shipped (DC §0, §9, §11.3).

But qdarte's kernel ships with three disqualifying defects the unified design must fix on day one — non-unique idempotency (F1), app-clock leases (F4), and the backoff-free lease-expiry loop (F6) — and its surrounding layers are the weakest part of the whole survey: three overlapping orchestration idioms (QO 7.1), two worker frameworks (QO §1.1), four runtime-control backends (QO 7.7), GETs that write (F3), and an API that secretly executes publish work (7.9). Diverse, conversely, got the operational shell right: DB-clock discipline, index-enforced idempotency with race-safe bulk enqueue, the scheduler-window claim table, environment interlocks, client-side settle idempotency, and a coherent single-idiom control plane (runs, dispatch leases, pool targets, generation fences) — while its engine carries a misnamed retry budget, a fixed backoff ladder, a broken hint parser, an FK-wedged and never-invoked cleanup, and zero bloat engineering.

Measured against prior art, both systems already implement the consensus core (SKIP LOCKED + lease + partial indexes + events + at-least-once + NOTIFY-as-optimization) and both fail the same two industry table stakes: nothing schedules maintenance reliably, and nothing exports metrics. The unified standard therefore starts from: qdarte's tables/CAS/policies/janitor + Diverse's DB clock/idempotency index/window table/control plane, plus the things *neither* has — unique-index idempotency wired into every enqueue path, a budget model that distinguishes failures from releases/expiries, pause/drain primitives, a scheduled tick that is not the claim hot path, DLQ-with-redrive affordances, FKs with ON DELETE, fillfactor, and a queue-depth/oldest-age metrics endpoint with alerting.
